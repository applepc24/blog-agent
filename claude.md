# 눕고싶은 개발자의 생존일지 - 블로그 자동화 에이전트

## 프로젝트 개요
비전공자 데이터 엔지니어 취준생이 운영하는 블로그(개발/데이터 + 다이어트/건강)의
콘텐츠 생산을 자동화하는 멀티 에이전트 시스템.

---

## 블로그 정보
- **블로그명**: 눕고싶은 취준생의 생존일지
- **플랫폼**: WordPress (자체 설치형, Linux 개인 서버)
- **주제**: 개발/데이터 (메인, 70%) + 다이어트/건강 (서브, 30%)
- **컨셉**: 혈압 경계선을 받은 비전공자 취준생의 개발 + 건강 생존 기록
- **목표**: 애드센스 수익화 (장기적, SEO 중심)

---

## 시스템 아키텍처

### 인터페이스
- **디스코드 봇**: 사용자(준석)와 편집장 에이전트가 대화하는 창구
  - 글감 자연어로 던지면 자동으로 장독대에 저장
  - 편집장이 주제 제안 및 최종 확인 요청

### 에이전트 구조 (LangGraph - Supervisor 패턴)

```
🏢 편집장 (Claude Opus) - 최소 호출
   · 준석이랑 디스코드로 대화
   · 글 방향 최종 결정
   · 최종 글 검토/승인만 담당
   · 장독대 글감 확인 후 준석에게 먼저 물어봄
         ↕
   [workflow.py - LangGraph Supervisor]
         ↕
┌─────────────────────────────────────┐
│  🔍 리서처      ✍️ 라이터    📊 SEO분석가  │
│  (Claude Haiku) (Claude Sonnet) (Claude Haiku) │
│  키워드/트렌드   글 초안 작성   제목 최적화  │
│  조사           경험 단락 정제  메타디스크립션│
│                               태그 선정    │
└─────────────────────────────────────┘
```
※ 조율자(Sonnet)는 규모 커지면 추가 예정, 현재는 workflow.py가 대신함

### 장독대 (글감 저장소)
- **DB**: SQLite
- **저장 방식**: 디스코드에서 자연어로 던지면 편집장이 자동 저장
- **활용**: 편집장이 새 글 주제 선정 시 장독대 먼저 확인 → 없으면 준석에게 물어봄

### 인프라
- **서버**: 개인 Linux 미니 PC (24시간 운영)
- **스케줄링**: cron
- **포스팅**: WordPress REST API 자동 업로드 (미구현, 서버 세팅 후)

## 모델별 역할 및 토큰 최적화 전략

| 에이전트 | 모델 | 이유 |
|---------|------|------|
| 편집장 | Claude Opus | 준석과 대화, 최종 검토만 - 호출 최소화 |
| 조율자 | Claude Sonnet | 중간 조율, 요약 후 Opus 전달 (미구현, 나중에 추가) |
| 라이터 | Claude Sonnet | 글 품질 중요 |
| 리서처 | Claude Haiku | 반복 작업, 비용 절약 |
| SEO분석가 | Claude Haiku | 반복 작업, 비용 절약 |
| 분석 에이전트 | Claude Haiku | 데이터 수집/분류, 비용 절약 |

**토큰 절약 핵심**: 하위 에이전트 결과를 Sonnet이 요약 → Opus에게 요약본만 전달

## 전체 워크플로우

### 글 생성 플로우
```
1. 편집장이 장독대 확인
2. 글감 있음 → "이거 어때요?" 준석에게 제안
   글감 없음 → "이번주 뭐 쓸까요?" 준석에게 먼저 물어봄
3. 준석이랑 디스코드에서 대화하며 주제 확정
4. workflow.py가 리서처/라이터/SEO분석가에게 지시
5. 리서처: 키워드 리서치
6. 라이터: SEO 최적화 초안 생성
7. SEO분석가: 제목/메타디스크립션/태그 최적화
8. 편집장이 준석에게 초안 공유: "올릴까요?"
9. 준석: "ㄱㄱ" or "수정해줘"
10. WordPress REST API로 자동 업로드 (서버 세팅 후)
```

### 피드백 루프 (매주)
```
1. 분석 에이전트가 Google Search Console API로 데이터 수집
2. 잘 된 글 / 안 된 글 분류
3. 조율자가 요약
4. 편집장이 준석에게 주간 리포트
   "이번주 가장 잘 된 글은 X예요, 다음 글은 이 방향 어때요?"
5. 준석 피드백 반영해서 다음 글 방향 결정
```

- 메타디스크립션 포함

## 주의사항
- Opus는 준석과 대화 + 최종 검토 시에만 호출 (토큰 최소화)
- 글 완전 자동화 X → 준석이 경험 한 줄 추가 + 최종 확인 필수
- 새 주제는 항상 준석에게 먼저 물어보고 결정
- 장독대 글감 우선 활용, 없을 때만 트렌드 기반 제안

## 세션 운영 규칙
- 세션 시작 시: `CLAUDE.md`, `PROGRESS.md` 읽고 현황 파악 후 응답
- 작업 단위 완료 시: `PROGRESS.md` 업데이트
- 컨텍스트 70% 도달 시: `/compact` 실행
